## О проекте

**qrmenus.kz** — B2B SaaS платформа, которая позволяет ресторанам Казахстана заменить бумажное меню на цифровое QR-меню. Ресторан регистрируется, загружает своё меню и получает QR-код. Гость сканирует код, листает меню на телефоне, делает заказ или вызывает официанта. Владелец видит все заказы в реальном времени, управляет меню через drag-and-drop и смотрит аналитику продаж на встроенном дашборде.

## Возможности

### Для персонала ресторана (Панель администратора)

- **Конструктор меню** — создание меню, категорий и блюд; сортировка drag-and-drop; изменения применяются мгновенно
- **Загрузка фотографий** — фото блюд хранятся в MinIO S3, отдаются через nginx с 30-дневным CDN-кешем
- **Мультиязычное меню** — публикация на русском, казахском и английском; AI-перевод через Gemini API
- **AI-функции** — автогенерация КБЖУ, описаний блюд и перевод всего меню одной кнопкой (с лимитами по тарифу)
- **Стоп-лист** — мгновенно скрыть блюдо без удаления
- **Аллергены** — стандартные коды аллергенов на каждом блюде; отображаются бейджами в гостевом меню
- **Журнал заказов** — входящие заказы в реальном времени со статусами (новый → готовится → готов → выдан)
- **Вызовы официанта** — гости вызывают официанта по номеру столика; персонал видит все активные вызовы
- **Telegram-интеграция** — подключить любой чат или группу; мгновенные уведомления о заказах и вызовах официанта без опроса сервера
- **Генератор QR-кода** — сгенерировать, просмотреть и скачать QR прямо из панели
- **Аналитика** — просмотры меню, просмотры блюд, разбивка по устройствам (мобильный/десктоп), топ блюд, графики по дням; глубина истории зависит от тарифа (7 / 30 / 90 дней)
- **Управление подпиской** — текущий тариф, история платежей, апгрейд плана (Freedom Pay готов к подключению)

### Для гостей (Публичное QR-меню)

- **Мгновенная загрузка** — меню отдаётся из Redis-кеша (TTL 300 с); без регистрации и приложений
- **Выбор языка** — переключение RU / KZ / EN на лету
- **Аллергены** — бейджи аллергенов на каждом блюде
- **Онлайн-заказ** — оформить заказ за столиком или предзаказ прямо с телефона
- **Вызов официанта** — кнопка → номер столика → уведомление персоналу
- **Анимации Framer Motion** — плавные переходы без лагов при прокрутке

### Технические

- **Мультитенантность** — полная изоляция ресторанов по `restaurant_id`; общая инфраструктура, разделённые данные
- **Верификация email** — двухшаговая регистрация (отправить код → подтвердить) через Resend
- **Жизненный цикл подписки** — 14-дневный триал создаётся автоматически при одобрении; ленивое истечение при проверке; отмена сохраняет доступ до конца периода
- **Проверка тарифов на уровне API** — `403 PLAN_LIMIT_REACHED` с подсказкой `upgrade_to`; фронтенд показывает модалку апгрейда
- **Web Push уведомления** — браузерные пуши с подписью VAPID (PWA-ready), без сторонних сервисов
- **Событийная аналитика** — просмотры публикуются в RabbitMQ; агрегируются ночным джобом; не блокируют запросы
- **Панель владельца платформы** — отдельная авторизация для Асанова: одобрение ресторанов, выручка, здоровье сервисов
- **Мягкое удаление** — рестораны, категории и блюда не удаляются физически (`deleted_at`); история сохраняется
- **Защита** — rate limiting в Nginx (вход 5 req/мин, API 30–60 req/мин), AI-квоты в Redis, memory limits на контейнерах

## Интересные технические решения

### 1. Двухшаговая регистрация с верификацией email

**Задача**: не допустить фиктивных аккаунтов и «висячих» данных в БД.

**Решение**: регистрация разбита на два эндпоинта. `POST /register-request` сохраняет временный код в таблицу `email_verifications` и отправляет его через Resend. `POST /register-verify` проверяет код, создаёт ресторан со статусом `pending` и асинхронно запускает активацию триала в billing-service (fire-and-forget с тремя попытками и экспоненциальным backoff). Запись в базе не появляется, пока email не подтверждён.

---

### 2. Ленивое истечение подписки

**Задача**: фоновые джобы, которые переводят подписки в статус `expired`, могут упасть тихо — тогда активные рестораны потеряют доступ.

**Решение**: ни один cron-джоб не меняет статус подписки самостоятельно. Вместо этого `billing-service` при каждом запросе `GET /internal/check/{restaurant_id}` сравнивает `trial_ends_at` / `current_period_end` с текущим временем и обновляет `status=expired` в той же транзакции. Источник истины — всегда строка в БД, а не заранее вычисленное состояние.

---

### 3. Событийная аналитика — без запросов на горячем пути

**Задача**: считать просмотры меню в реальном времени при каждом запросе страницы значит нагружать PostgreSQL без необходимости.

**Решение**: `menu-service` публикует лёгкие события `menu_view` и `item_view` в RabbitMQ TOPIC exchange (`menu_events`). `analytics-service` потребляет события асинхронно и записывает их в `analytics.analytics_events`. Джоб в 01:00 UTC агрегирует события прошлого дня в `analytics.daily_aggregates` (общие просмотры, разбивка по устройствам, пиковый час, топ блюд). Аналитические запросы никогда не затрагивают горячий путь запроса.

---

### 4. Флаг `is_deleted` в аналитике заказов

**Задача**: блюдо, популярное в прошлом месяце, могло быть удалено сегодня. Аналитика должна показывать исторические данные без ошибки «внешний ключ не найден».

**Решение**: заказы хранятся как JSONB-снимок корзины в момент покупки. Запрос топ-блюд по заказам читает данные напрямую из `orders.items` JSONB и выставляет флаг `is_deleted`, если блюдо больше не существует в таблице `items`. В интерфейсе такое блюдо отображается с бейджем «Удалённое блюдо».

---

### 5. Мягкое удаление + частичный уникальный индекс для повторной регистрации

**Задача**: ресторан может отменить подписку и захотеть зарегистрироваться заново с тем же email. Обычный `UNIQUE`-индекс заблокирует это навсегда.

**Решение**: рестораны мягко удаляются (`deleted_at IS NOT NULL`). Частичный уникальный индекс `UNIQUE (email) WHERE deleted_at IS NULL` обеспечивает уникальность только среди активных записей — тот же email можно использовать повторно после удаления без костылей на уровне приложения.

---

### 6. AI Guard — rate limiting и дневные квоты через Redis

**Задача**: запросы к Gemini API стоят денег. Один «агрессивный» клиент может исчерпать дневной лимит.

**Решение**: каждый AI-эндпоинт проходит через `ai_guard.py`, который ведёт два независимых счётчика в Redis на каждый ресторан:

- **Скользящее окно** (5 запросов/час): `INCR ai:{restaurant_id}:{endpoint}:{hour}` + `EXPIRE 3600`.
- **Дневная квота** (10 запросов/день): `INCR ai:{restaurant_id}:{endpoint}:{date}` + `EXPIRE 86400`.

Если хотя бы один счётчик превышен — эндпоинт возвращает `429` до вызова Gemini. Каждый AI-запрос логируется структурированно: `restaurant_id`, название эндпоинта, примерное количество токенов, метка времени.

---

### 7. Telegram-бот — несколько получателей с автоочисткой

**Задача**: рестораны хотят получать уведомления в несколько чатов (кухня, менеджер, владелец). Если бот выгнан из группы, «мёртвые» получатели накапливаются.

**Решение**: администратор генерирует одноразовый 6-значный код с TTL 10 минут. Сотрудник отправляет код в любой чат с ботом. Вебхук активирует код и создаёт запись `TelegramRecipient` с `chat_id` и понятным `label`. Поддерживается несколько получателей на один ресторан. Когда бот получает `my_chat_member` с типом `kicked` или `banned` — получатель автоматически удаляется без ручного вмешательства.

---

### 8. Web Push — самостоятельный VAPID без сторонних сервисов

**Задача**: пуш-уведомления через Pusher или Firebase — это дополнительные расходы и зависимость от стороннего сервиса.

**Решение**: `notification-service` управляет собственной VAPID-парой ключей. Публичный ключ передаётся фронтенду; приватный ключ подписывает Web Push payload на сервере через `pywebpush`. Эндпоинт `/internal/send-push` закрыт на уровне Nginx для публичного трафика — доставка пушей всегда инициируется только изнутри платформы. Стандартный браузерный push без единой внешней SaaS-зависимости.

---

### 9. Swap-файл + последовательные Docker-сборки для VPS с 2 ГБ RAM

**Задача**: параллельная сборка Docker-образов исчерпывает RAM на дешёвом VPS и роняет deploy через OOM Kill.

**Решение**: в продакшн-деплое используется swap-файл на 2 ГБ, а сборка сервисов выполняется последовательно (`--no-cache` по одному), а не параллельно. На каждый контейнер выставлены memory limits (128–512 МБ), чтобы сумма RAM в рабочем режиме не приближалась к физической + swap.

---

### 10. Инвалидация кеша через RabbitMQ

**Задача**: при любом изменении блюда/категории/меню нужно сбросить Redis-ключи — но `admin-service` не должен владеть ключами, которыми управляет `menu-service`.

**Решение**: после каждой записи в `admin-service` в RabbitMQ публикуется `menu_slug`. `menu-service` (который владеет кешем) потребляет сообщение и удаляет нужные ключи (`menu:{slug}:*`). Паттерн разделяет запись и управление кешем без прямой связи между сервисами.

## Бизнес-контекст

**qrmenus.kz** разработан и выведен в продакшн как сольный проект. Платформа работает и доступна по адресу [qrmenus.kz](https://qrmenus.kz).

**Исследование рынка**: до и в процессе разработки я провёл 15+ структурированных интервью с владельцами ресторанов и кафе в Шымкенте. Главный вывод — традиционный HoReCa-сегмент требует значительно более длинного цикла продаж, чем обычный B2B SaaS. Владельцы открыты к идее, но нужна живая демонстрация, рекомендация от знакомых и пробный период. Бумажное меню — глубокая привычка, а воспринимаемая стоимость перехода (обучение персонала, печать QR, первоначальная настройка) — главный барьер, а не цена.

**Текущий статус**: платформа в пре-revenue фазе. Идёт активный поиск первого платящего клиента. Технический продукт готов к работе; узкое место — продажи, не разработка.

**Что намеренно отложено**: интеграция Freedom Pay реализована как заглушка (`PAYMENT_ENABLED = False`), чтобы не обрабатывать реальные карточные данные до появления первого платящего клиента. Вебхуки Kaspi и CloudPayments подключены и готовы к активации.

## Что я узнал

**Технически**

1. **Async Python требует дисциплины.** SQLAlchemy 2.0 async + asyncpg мощный инструмент, но управление жизненным циклом сессии (избежать «session closed» при конкурентной нагрузке) требует явного скоупинга в каждом роуте — магической переменной контекста не существует.

2. **TOPIC exchange в RabbitMQ — правильный примитив для мультитенантной маршрутизации событий.** Ключ `{event_type}.{restaurant_id}` позволяет подписаться на события одного ресторана или на все сразу одним wildcard-биндингом — без создания отдельной очереди на каждого клиента.

3. **Redis как хранилище rate-limit — дёшево и атомарно.** `INCR` + `EXPIRE` на составном ключе — это скользящее окно без внешних зависимостей. Собственный guard занял ~50 строк кода и убрал одну библиотечную зависимость.

4. **Мягкое удаление создаёт скрытую сложность в запросах.** Каждый запрос должен содержать `WHERE deleted_at IS NULL`. Пропустить это в одном месте — значит молча показать удалённые данные. Частичные уникальные индексы — чистое решение для повторной регистрации, но паттерн требует документирования и дисциплины во всех сервисах.

5. **pnpm workspaces делают общую библиотеку компонентов практичной.** `@qrmenu/ui` импортируется напрямую во всех трёх SPA без публикации в npm. Hot reload работает между пакетами. Стоимость настройки окупается единообразием дизайна.

**Бизнесово**

6. **Сначала проверяй модель продаж, потом пиши код.** Продукт технически верный — но «технически верный» не сокращает HoReCa-цикл продаж. Более простой MVP (хотя бы Google Forms + QR вручную) позволил бы понять динамику продаж быстрее и с меньшими вложениями.

7. **Платёжная инфраструктура Казахстана фрагментирована.** Три разных шлюза (Kaspi, CloudPayments, Freedom Pay) покрывают пересекающиеся, но разные сегменты рынка. Поддержать все три — реальное продуктовое требование, а не избыточная инженерия. Это единственный момент, где пользовательские интервью напрямую повлияли на архитектуру.
